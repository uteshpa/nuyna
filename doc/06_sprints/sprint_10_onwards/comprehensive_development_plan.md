## 1. エグゼクティブサマリー

本計画は、QA評価レポート（総合スコア58/100）で明らかになった**致命的な品質問題**と、**テスト環境の深刻な制約**という2つの課題を克服し、nuynaを市場に投入可能な製品へと再生させるための総合的な開発計画です。

**基本方針:**
- **品質第一:** 新機能開発を凍結し、P0/P1レベルの欠陥修正に全リソースを集中します。
- **段階的アプローチ:** 8週間を3つのスプリントに分け、**「①基盤修正 → ②性能改善 → ③リリース準備」**の順で品質を段階的に引き上げます。
- **リスク管理:** Android未テスト問題に対し、**「クラウドテスト → 実機検証 → 段階的ロールアウト」**の3層防御戦略で対応します。

**最終目標:** 8週間後に、ユーザーの信頼を回復し、自信を持ってマネタイズ（300円での販売）を開始できる品質レベルを達成します。

## 2. 現状の課題と市場分析

### 2.1. プロダクトの品質課題

現在のnuynaは、以下の重大な問題を抱えています。

| 優先度 | 問題点 | 内容 |
| :--- | :--- | :--- |
| **P0** | **iOSメタデータ削除機能不全** | プライバシー保護というコアバリューを根底から覆す致命的欠陥。 |
| **P1** | **UI操作系の混乱** | ボタンとスイッチが混在し、ユーザー体験を著しく損なっている。 |
| **P1** | **パフォーマンス目標未達** | 現行端末での目標達成が絶望的であり、実測と目標の再設定が必須。 |
| **P1** | **テストの欠如** | 統合・E2Eテストがなく、品質が保証されていない。 |

### 2.2. テスト環境の制約とマーケティング的視点

**テスト端末:** iPhone 11 Pro Max (2019年発売)

- **市場代表性:** **極めて低い。** 市場シェアは2-3%程度で、大多数のユーザー環境を代表していません。
- **性能:** 現行のiPhone 15 Proと比較して、**CPU性能は約半分以下**です。

**マーケティング上の結論:**
> iPhone 11 Pro Maxでのテスト結果は、あくまで**「最低動作保証ライン」**と捉えるべきです。この端末で快適に動作しない場合、市場の大多数のユーザーにとっても体験は悪化します。パフォーマンス目標を設定する際は、この端末での測定値に**2.2倍程度の性能向上係数**を適用し、現行端末での体験を推定する必要があります。

### 2.3. Android未テストのリスク

**グローバル市場の72%を占めるAndroidプラットフォームが完全にブラックボックス**であり、このままではリリースは不可能です。最終スプリントまで実機テストができないという制約を乗り越えるため、特別な戦略が求められます。

## 3. 開発スプリント計画（8週間）

### Sprint 1: 基盤修正スプリント (2週間)

**ゴール:** 致命的な欠陥（P0, P1）を修正し、プロダクトの信頼性を回復する。

| 優先度 | タスク |
| :--- | :--- |
| **P0** | iOSメタデータ削除問題の完全解決（カスタムPlatform Channel実装） |
| **P1** | UI/UXの抜本的改善（操作系統一、ワンタップ処理実装） |
| **P1** | E2Eテストの導入（Firebase Test Lab） |
| **P2** | 未使用の依存パッケージ（4つ）の削除 |

### Sprint 2: パフォーマンス改善とテスト強化スプリント (2週間)

**ゴール:** パフォーマンスを現実的なレベルまで改善し、Androidの初期テストを開始する。

| 優先度 | タスク |
| :--- | :--- |
| **P1** | パフォーマンス測定と目標再設定（ベンチマーク係数適用） |
| **P1** | パフォーマンス改善（ボトルネック分析と実装） |
| **P1** | **Android初期テスト（クラウドテスト）** |
| **P2** | 統合テストの実装 |

### Sprint 3: リリース準備とAndroid対応スプリント (3週間)

**ゴール:** iOS版のリリース準備を完了し、Android版の品質をリリース可能レベルまで引き上げる。

| 優先度 | タスク |
| :--- | :--- |
| **P0** | **Android実機テスト（調達した実機を使用）** |
| **P1** | Android固有のバグ修正 |
| **P1** | ベータテストの実施（iOS/Android） |
| **P2** | ストア申請準備、クラッシュレポート導入 |

## 4. Android未テスト問題への3層防御戦略

| 防御層 | タイミング | アクション | 目的 |
| :--- | :--- | :--- | :--- |
| **第1層** | **Sprint 2** | **クラウドテスト (Firebase Test Lab)** | 開発中にクラッシュや基本動作不良を自動検知 |
| **第2層** | **Sprint 3** | **実機検証 + ベータテスト** | パフォーマンスやハードウェア固有の問題を人間が検証 |
| **第3層** | **リリース後** | **段階的ロールアウト + 監視** | 万が一の問題発生時に影響を最小限に抑える |

この戦略により、実機がない序盤のリスクを最小化し、安全にAndroid市場への参入を目指します。
